# GSoC 2026 Project Journal: RoaringBitmap Functions for Apache Fluss

This file serves as the central hub for my Google Summer of Code 2026 project. Instead of a static document, this is a living journal to track my progress, challenges, and learnings as I implement native RoaringBitmap functions in Apache Fluss.

## My Mission

My goal is to make powerful bitmap-based analytics a seamless, out-of-the-box experience for Apache Fluss users. By implementing and registering a core set of RoaringBitmap functions directly into the Flink SQL layer, I will remove the need for custom UDFs and reduce the friction of performing high-performance, real-time analytics.

## Project Log

### Week 2 (Coding Period: May 27 - June 2)
*   **What I did:** Made significant progress on the core implementation for FIP-37.
    *   **[MERGED]** My first PR, **[#3319](https://github.com/apache/fluss/pull/3319)**, was successfully merged! This PR lays the foundational infrastructure for all the new bitmap functions, including serialization utilities (`BitmapUtils`, `RoaringBitmapSerializer`) and a base class for aggregate functions.
    *   **[IN REVIEW]** Submitted the next PR, **[#3398](https://github.com/apache/fluss/pull/3398)**, which implements the first three aggregate functions: `rb_build_agg`, `rb_or_agg`, and `rb_and_agg`. This PR also handles their registration in the `FlussCatalog`.
*   **Challenges I faced:** The initial review for PR #3398 highlighted the need for a full end-to-end SQL ITCase to validate the Flink planner integration, which was more complex than simple unit tests. I've since added this to the PR.
*   **What I learned:** I gained a much deeper understanding of Flink's type system, the importance of custom `TypeSerializer`s for complex types like RoaringBitmap, and how the Flink planner uses `FunctionHint` to handle UDFs. The review process was incredibly valuable for learning how to build robust, production-ready code.

### Week 1 (Community Bonding: May 13 - May 26)
*   **What I did:** Finalized project scope with mentors, studied the existing `AggregationMergeEngine` and `FlussCatalog` code, and set up my development environment.
*   **Challenges I faced:** Understanding the distinction between server-side aggregation and the Flink SQL function layer.
*   **What I learned:** The importance of FIP-37's scope being limited to the SQL function layer to avoid client-server compatibility issues.
