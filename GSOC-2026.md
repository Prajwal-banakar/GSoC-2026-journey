# GSoC 2026 Project Journal: RoaringBitmap Functions for Apache Fluss

This file serves as the central hub for my Google Summer of Code 2026 project. Instead of a static document, this is a living journal to track my progress, challenges, and learnings as I implement native RoaringBitmap functions in Apache Fluss.

## My Mission

My goal is to make powerful bitmap-based analytics a seamless, out-of-the-box experience for Apache Fluss users. By implementing and registering a core set of RoaringBitmap functions directly into the Flink SQL layer, I will remove the need for custom UDFs and reduce the friction of performing high-performance, real-time analytics.

*   **PRs:** [#3470](https://github.com/apache/fluss/pull/3470), [#3497](https://github.com/apache/fluss/pull/3497)

### Week 5 (Coding Period: June 10 - June 16)
*   **What I did:** Submitted the pull request for the Phase 1 scalar functions. This PR includes the implementation of `rb_cardinality`, `rb_build`, `rb_contains`, `rb_to_array`, `rb_or`, and `rb_and`.
*   **PR:** [#3492](https://github.com/apache/fluss/pull/3492)
*   **Challenges I faced:** Ensuring the new functions were correctly registered in the `FlussCatalog` and that the end-to-end ITCase covered all the new scalar functions.
*   **What I learned:** The value of reusing test infrastructure. I was able to extend the `RbFunctionsCatalogITCase` from the previous PR to cover the new scalar functions.

### Week 4 (Coding Period: June 3 - June 9)
*   **What I did:** Started working on the implementation of the Phase 1 scalar functions. This involved creating the individual `ScalarFunction` classes and ensuring they correctly handled serialization and deserialization of the RoaringBitmap format.
*   **What I learned:** The difference between `AggregateFunction` and `ScalarFunction` in Flink is not just about the number of inputs, but also about how they manage state and interact with the Flink planner.

### Week 3 (Coding Period: May 27 - June 2)
*   **What I did:** A major milestone week!
    *   **[MERGED]** My second major PR, **[#3398](https://github.com/apache/fluss/pull/3398)**, was successfully merged! This PR implements the first three aggregate functions (`rb_build_agg`, `rb_or_agg`, `rb_and_agg`) and registers them in the `FlussCatalog`.
    *   **[MERGED]** My first PR, **[#3319](https://github.com/apache/fluss/pull/3319)**, which laid the foundational infrastructure, was also merged this week.
*   **Challenges I faced:** The review process for PR #3398 was incredibly thorough and highlighted the need for a full end-to-end SQL ITCase. Implementing this was a great learning experience.
*   **What I learned:** The importance of robust integration testing cannot be overstated. The feedback from my mentors on the catalog registration strategy was also invaluable and gave me confidence in the design. It's been a great learning experience, and I’m really grateful for the guidance.

### Week 1-2 (Community Bonding & Initial Coding: May 13 - May 26)
*   **What I did:** Finalized project scope with mentors, studied the existing `AggregationMergeEngine` and `FlussCatalog` code, and set up my development environment. The extensive discussion on the mailing list was crucial in refining the scope of FIP-37.
*   **Challenges I faced:** Understanding the distinction between server-side aggregation and the Flink SQL function layer. The feedback from Jark Wu was particularly helpful in clarifying that we should defer the `BITMAP` type and pushdown optimization.
*   **What I learned:** The importance of community feedback in shaping a proposal. The initial proposal was much broader, but the feedback from the community helped to focus it on a deliverable and high-impact set of features.
